## 一、产品定位

### 1.1 一句话定义

**Learning Diagnosis 是一个把"培训请求"转化为"干预决策"的证据引擎。**

它不是帮讲师更快地接培训需求，而是帮讲师从"被动接单的内容生产者"升级为"对绩效问题做判断的专业顾问"。

### 1.2 目标用户

**企业培训讲师 / 学习设计师**——在企业内部或作为外部顾问，负责接收业务方的培训需求并设计学习干预方案的专业人士。

用户典型画像：
- 有 3-10 年培训经验，具备基本的教学设计能力
- 日常面对的核心挑战不是"做不出课"，而是"接到的需求本身就有问题"
- 需要跟业务方、HR、管理层多方沟通，说服力和专业判断力比制作速度更重要
- 时间碎片化：一个诊断项目可能跨越数天，中间穿插访谈、问卷、会议

### 1.3 核心问题

业务方口中的"培训需求"往往不是真正的问题，而是问题被组织叙述后的样子。它可能是对问题的粗糙命名、对责任的外包、对"先做点什么"的象征性动作。

现有工具要么只做"需求收集表单"（无法识别叙事偏差），要么直接跳到"生成课纲"（跳过了诊断）。两种方式都导致同一个结果：更快地产出错误方案。

### 1.4 产品价值主张

| 维度 | 现状 | 本产品提供的 |
|------|------|-------------|
| 需求理解 | 业务方说什么就接什么 | 自动解构叙事、生成竞争性假设 |
| 诊断维度 | 只看"学员缺什么" | 覆盖业务-绩效-能力-环境-管理-动机六层 |
| 分析粒度 | 围绕课程主题 | 围绕关键工作情境（Role × Moment × Behavior × Friction） |
| 信息收集 | 填完所有字段才能进行下一步 | 证据充分性驱动，够了就停 |
| 输出 | 课程大纲 | 干预判决书（含"不做培训"选项 + 利益相关方沟通方案） |
| 评估 | 课程结束后才想 | 诊断阶段就定义行为指标和 baseline |

## 二、核心设计原则

以下七条原则来自领域深度分析，贯穿所有产品决策。

| # | 原则 | 产品设计含义 |
|---|------|-------------|
| 1 | 需求是假设，不是事实 | 产品第一职责不是接收需求，而是解构需求 |
| 2 | 学习问题镶嵌在绩效系统中 | 诊断模型必须覆盖六层，不能只看能力 |
| 3 | 最小可干预单位是关键工作情境，不是课程主题 | 产品围绕 critical moments 建模，不围绕 topics |
| 4 | 诊断是假设筛选，不是信息收集 | 产品管理的是竞争假设、置信度和证据缺口 |
| 5 | 不做培训是正当输出 | "不给课"必须是一个有尊严的产出路径 |
| 6 | 评估从诊断阶段开始 | 行为指标和 baseline 在诊断中原生定义 |
| 7 | 讲师的核心价值是判断力 | 产品放大的是判断能力，不是生产速度 |

```

│                          │                                      │

│  - 支持粘贴长文本         │  - 六层诊断板                        │
```

### 3.2 面板宽度比例

- 默认状态：左 45% / 右 55%（面板内容较丰富，给多一些空间）

- 面板可折叠：讲师想专注对话时可以收起右面板

## 四、右侧面板：诊断结构

### 4.1 面板渐进生长机制

面板不是一开始就把所有区域全部展示，而是随对话推进逐步"长出来"。这降低了初始认知负荷，同时让讲师感受到"信息在逐步成型"的进展感。

#### 阶段一：种子（COLD_START 之后）

Agent 完成第一次问题转译后，面板上出现：

```
┌─────────────────────────────────────┐
│  📋 问题转译                         │
│  ┌─────────────────────────────────┐│
│  │ 表层请求：给店长做执行力培训      ││
│  │ 业务意图：门店活动落地不一致      ││
│  │ 行为假设：店长未按标准完成跟进    ││
│  │ 成因假设：责任心不足 / 管理弱    ││
│  │                                 ││
│  │ ⚠️ 叙事偏差风险：               ││
│  │ 把"执行差"过早解释为"态度差"     ││
│  └─────────────────────────────────┘│
│                                     │
│  💡 竞争假设                         │
│  ┌─────────┐ ┌─────────┐ ┌────────┐│
│  │ H1      │ │ H2      │ │ H3     ││
│  │ 不知道  │ │ 现场    │ │ 经理不 ││
│  │ 标准流程│ │ 顾不上  │ │ 做跟进 ││
│  │ ━━░░░░ │ │ ━━░░░░ │ │ ━░░░░░ ││
│  │ 能力层  │ │ 环境层  │ │ 管理层 ││
│  └─────────┘ └─────────┘ └────────┘│
│                                     │
│  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌ │
│  还有 4 个维度尚未涉及 ▸             │
└─────────────────────────────────────┘
```

此时只有两个区块。面板简洁、轻量，讲师不会觉得"要填满一个大表"。

#### 阶段二：分枝（对话深入后）

当对话触及新的维度时，对应区块才出现：

- 讲师提到了业务目标/KPI → **Business 层**从六层板中冒出
- 讲师描述了具体工作场景 → **关键时刻**区出现
- 讲师提到了经理的做法 → **Management 层**冒出
- 以此类推

每个新出现的区块有一个简短的"入场动画"（淡入+轻微展开），让讲师注意到面板在成长。

#### 阶段三：成熟（证据逐渐充分）

当多数层都有了信息，面板顶部出现**全局证据充分度指示器**：

| 状态 | 含义 | 视觉 |
|------|------|------|
| 🔴 证据不足 | 关键判断仍缺证据，不建议做设计决策 | 红色，提示"建议继续诊断" |
| 🟡 初步可判 | 主要假设有初步证据，可出初步方案但需标注风险 | 黄色，提示"可生成初步建议" |
| 🟢 可以决策 | 核心假设已有充分证据，可进入干预设计 | 绿色，提示"可进入下一阶段" |

同时，底部出现**干预建议区**（在 DECISION_READY 状态下由 Agent 生成）。

### 4.2 面板区块详细定义

#### 区块 A：问题转译卡

| 字段 | 说明 | 可编辑 |
|------|------|--------|
| 表层请求 | 业务方原话或摘要 | ✅ |
| 业务意图 | Agent 推断的真实关切 | ✅ |
| 行为假设 | 业务方认为哪些行为没发生 | ✅ |
| 成因假设 | 业务方默认的原因 | ✅ |
| 叙事偏差风险 | Agent 标记的可能偏差 | ❌（Agent 生成，讲师可隐藏但不可编辑） |

讲师编辑任何字段后，系统消息通知 Agent，Agent 在下一轮回复中基于新内容调整。

#### 区块 B：竞争假设区

每个假设呈现为一张**卡片**，包含：

| 元素 | 说明 | 可交互 |
|------|------|--------|
| 假设内容 | 一句话描述 | ✅ 可编辑 |
| 所属层级 | business / performance / capability / environment / management / motivation | ✅ 可修改 |
| 置信度 | 可视化进度条 (none → low → medium → high) | ✅ 可拖拽调整 |
| 支持证据 | 列表，每条标明来源 | ✅ 可增删 |
| 反对证据 | 列表 | ✅ 可增删 |
| 状态 | 待验证 / 证据支持 / 已削弱 / 已排除 | ✅ 可切换 |

卡片交互：
- **Hover**：显示操作菜单（深入追问 / 寻找反面证据 / 排除此假设）
- **点击"深入追问"**：对话区自动注入上下文，Agent 围绕该假设展开
- **拖拽**：可调整假设的排列顺序（优先级）
- **标记"已排除"**：卡片视觉变灰，移到区域底部但不消失（可恢复）

讲师也可以**手动添加新假设卡片**（输入内容、选择层级），Agent 会在后续对话中将其纳入考量。

#### 区块 C：六层诊断板

六个层级，每层显示为一个可展开的行：

```
┌─────────────────────────────────────────┐
│ ▸ 🏢 Business 层      ━━━━━━░░░░  部分  │
│ ▸ 📊 Performance 层   ━━━━━━━━░░  充分  │
│ ▾ 🧠 Capability 层    ━━░░░░░░░░  不足  │
│   ┌─────────────────────────────────┐   │
│   │ 摘要：一线销售缺乏探因提问技巧， │   │
│   │ 但产品知识测试得分正常。          │   │
│   │ 来源：讲师访谈记录 (3/12)        │   │
│   │                        [编辑]    │   │
│   └─────────────────────────────────┘   │
│ ▸ 🔧 Environment 层   ░░░░░░░░░░  空白  │
│ ▸ 👤 Management 层    ━░░░░░░░░░  不足  │
│ ▸ 💭 Motivation 层    ░░░░░░░░░░  空白  │
└─────────────────────────────────────────┘
```

每层展开后显示：
- **摘要**（Agent 生成，讲师可编辑）
- **证据来源**（标注来自哪次对话、哪份材料）
- **状态**：空白 / 部分 / 充分
- 操作按钮：**编辑** / **深入追问此层**

空白和不足的层级视觉上较淡，暗示"这里还需要信息"，但不强制。

#### 区块 D：关键时刻列表

当对话中识别出具体工作情境时出现。每条格式：

```
┌──────────────────────────────────────────────┐
│ 🎯 关键时刻 #1                                │
│ 角色：区域销售经理                              │
│ 时刻：客户第一次提出价格异议时                   │
│ 理想行为：先探因 → 重述价值 → 让步边界管理       │
│ 当前阻力：不会探因；没有话术模板；害怕丢单       │
│                                   [编辑] [追问] │
└──────────────────────────────────────────────┘
```

讲师可以：
- 编辑每个字段
- 添加新的关键时刻
- 删除不相关的
- 点"追问"让 Agent 围绕该时刻深入分析

#### 区块 E：评估锚点

在诊断过程中逐步积累，最终成为评估方案的基础：

| 字段 | 说明 | 可编辑 |
|------|------|--------|
| 行为指标 | 可观察的行为变化，如"一线销售在客户提出异议后主动探因的比例" | ✅ |
| 当前基线 | 粗略估计即可，如"目前大约 20% 的销售会主动探因" | ✅ |
| 观察者 | 谁能看到这个变化，如"直属经理、区域巡查" | ✅ |
| 观察时机 | 什么时候能看到，如"训后 2 周的真实客户拜访" | ✅ |

#### 区块 F：干预建议（DECISION_READY 时出现）

2-3 个并列的方案卡片：

```
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ 方案 A          │ │ 方案 B          │ │ 方案 C          │
│ ──────────────  │ │ ──────────────  │ │ ──────────────  │
│ 训练为主 +      │ │ 先做 Job Aid,   │ │ ⚠️ 暂不建议培训  │
│ 经理 follow-up  │ │ 再决定是否培训  │ │ 先调整激励机制   │
│                 │ │                 │ │                 │
│ 适用条件：...   │ │ 适用条件：...   │ │ 适用条件：...   │
│ 风险：...       │ │ 风险：...       │ │ 风险：...       │
│ 评估方式：...   │ │ 评估方式：...   │ │ 评估方式：...   │
│                 │ │                 │ │                 │
│   [选择此方案]  │ │   [选择此方案]  │ │   [选择此方案]  │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

当讲师选择包含"不建议培训"的方案时，系统额外生成一份**利益相关方沟通话术框架**，帮助讲师向业务方专业地传达这个判断。

## 五、双面板数据流

### 5.1 数据流方向

三条路径构成闭环：

```
          ① 对话 → 面板
    ┌──────────────────────────┐
    │    讲师发言 → Agent 回复  │
    │    Agent 输出 JSON 更新   │──────→ 面板渲染更新
    └──────────────────────────┘

          ② 面板 → 对话
    ┌──────────────────────────┐
    │  讲师点击面板元素的       │
    │  "深入追问"按钮          │──────→ 对话区注入上下文提问
    └──────────────────────────┘       Agent 针对该元素回复

          ③ 面板自身编辑
    ┌──────────────────────────┐
    │  讲师直接编辑面板字段     │──────→ 面板立即更新
    │  或调整假设状态/置信度    │       + 通知 Agent（见 5.2）
    └──────────────────────────┘
```

### 5.2 面板编辑 → Agent 通知策略

讲师编辑面板内容时，Agent 需要知道。但通知方式取决于编辑的**影响程度**：

#### 轻量编辑（沉默同步 + 系统消息）

包括：修改措辞、调整描述文字、增加证据条目、调整卡片排序。

处理方式：
1. 面板即时更新
2. 对话区出现一条灰色系统消息："讲师修改了 [H2] 的描述"
3. Agent 下次回复时自动基于新状态，但不特别提及这次编辑

#### 方向性编辑（主动回应）

包括：排除/恢复一个假设、把某层状态从"不足"改为"充分"、删除一个关键时刻、修改核心行为指标。

处理方式：
1. 面板即时更新
2. 对话区出现系统消息
3. **Agent 主动生成一条回复**，评估这个变更对诊断方向的影响：

> "我注意到你把 H3（经理不做跟进）标为了已排除。这样的话，管理层面目前就没有活跃假设了。你是已经有了确认信息，还是觉得这个方向优先级低？这会影响我们后续要不要建议经理带教环节。"

判定规则：当编辑导致**任何活跃假设被排除、任何诊断层状态变化、或干预建议的前提条件被动摇时**，触发主动回应。

### 5.3 对话 → 面板的更新机制

Agent 每次回复都携带一个 `<diagnosis_update>` JSON 块（详见 Agent System Prompt）。前端接收到这个 JSON 后：

1. **Diff 对比**：与当前面板状态对比，识别变更字段
2. **动画更新**：变更的字段有视觉高亮（如短暂的背景色闪烁），让讲师注意到哪里变了
3. **冲突检测**：如果 Agent 更新的字段恰好是讲师刚刚手动编辑的，弹出提示："Agent 对此也有新判断，要查看吗？"讲师可选择保留自己的版本或查看 Agent 的版本

## 六、交互流程

### 6.1 冷启动

讲师第一次打开工具（或创建新诊断项目）时：

**左面板**：显示 Agent 开场白：

> "把你收到的培训需求贴过来吧——微信消息、邮件、会议纪要、或者你自己的一句话描述都行。我先帮你拆一下。"

**右面板**：极简空白状态，只有一句引导文字和品牌图形，不显示任何空表单框架。

**输入方式**：
- 文本输入框（支持长文本粘贴）
- 附件上传按钮（支持 .txt, .docx, .pdf, 图片）
- 快捷粘贴（Ctrl+V 即可直接粘贴，无需点击）

讲师提交输入后，Agent 执行"问题转译"，右面板进入种子阶段。**这是产品的"Aha moment"**——讲师第一次看到自己的模糊需求被结构化拆解。

### 6.2 主对话流

对话过程中的标准循环：

```
讲师发言 / 粘贴素材 / 回答问题
         ↓
Agent 回复（自然语言 + diagnosis_update JSON）
         ↓
面板更新（变更字段高亮）
         ↓
讲师可选择：
  a) 继续对话
  b) 编辑面板
  c) 点击面板元素发起追问
  d) 查看全局状态，决定是否继续
```

### 6.3 从面板发起追问

讲师在面板上 hover 任何可交互元素（假设卡片、诊断层、关键时刻），出现操作菜单：

| 操作 | 说明 | 效果 |
|------|------|------|
| 深入追问 | 让 Agent 围绕这个元素展开分析 | 对话区注入上下文消息，Agent 回复 |
| 寻找反面证据 | 让 Agent 挑战当前判断 | Agent 主动提供反面论据 |
| 帮我设计验证方式 | 让 Agent 提供取证工具包 | Agent 生成迷你访谈提纲/问卷/观察清单 |
| 排除 / 恢复 | 标记假设为已排除或恢复 | 面板更新 + Agent 主动回应（方向性编辑） |

点击操作后，对话区自动生成一条带上下文的消息（讲师可编辑后发送，也可直接发送）：

> "[关于 H3: 经理不做跟进检查] 帮我想想怎么验证这个假设——我下周要跟区域经理开会，有什么问题可以顺便问的？"

### 6.4 诊断完成 → 输出

当全局状态指示器变为 🟢，或讲师认为信息已足够时：

1. Agent 主动提出可以生成干预建议（也可由讲师手动触发）
2. Agent 生成 2-3 个干预方案，面板底部出现**干预建议区**
3. 讲师选择方案后，系统生成一份**干预判决书**，包含：

| 章节 | 内容 |
|------|------|
| 问题定义 | 从原始需求到结构化问题的转译 |
| 诊断发现 | 六层诊断结果摘要 |
| 关键时刻 | 识别出的核心工作情境 |
| 假设与证据 | 各假设的最终状态和依据 |
| 干预建议 | 选定的方案 + 替代方案 |
| 评估方案 | 行为指标、基线、观察者、时机 |
| 风险与假设 | 哪些判断仍待验证 |
| 利益相关方沟通建议 | 话术框架（尤其当建议"不做培训"时） |

这份判决书可导出为文档，也是下一阶段（课程设计）的输入。

## 七、撤销与历史

### 7.1 快照时间线模型

面板不采用逐操作的 Undo/Redo 栈，而是采用**快照时间线**。

**触发快照的时机**（自动保存）：
- 每次 Agent 回复导致面板更新
- 每次讲师完成一次有意义的面板编辑（防抖：连续编辑在 3 秒内只存一次快照）
- 讲师手动点击"保存当前状态"

**时间线 UI**：

面板顶部有一个可展开的时间线滑块：

```
┌─────────────────────────────────────────────┐
│  ◀ ──●────●──●──────●──●────●──●── ▶        │
│       T1   T2 T3    T4 T5   T6 T7(当前)     │
│  每个节点显示时间戳 + 变更摘要预览             │
└─────────────────────────────────────────────┘
```

讲师可以：
- **点击任意节点**：预览该时刻的面板状态（只读）
- **恢复到某节点**：回退面板到该历史状态（会创建一个新快照标记"从 T4 恢复"）
- **对比两个节点**：查看两个时间点之间面板的变化（diff 视图）

### 7.2 恢复不影响对话

面板恢复到历史状态时，左侧对话历史不会被删除或回退。Agent 在下一轮回复中会感知到面板状态的变化（因为当前面板状态始终是 Agent 的上下文之一），并据此调整对话。

## 八、跨会话 / 异步诊断

### 8.1 诊断项目

核心概念不是"对话"而是**诊断项目**。

- 讲师创建一个诊断项目，它有名称、创建时间、状态（进行中 / 已完成）
- 一个项目内可以包含**多次对话会话**
- 右面板的状态**跨会话持久化**

### 8.2 回归体验

讲师隔天重新打开一个进行中的项目时：

- 右面板**完整恢复**到上次的状态
- 对话区显示**上次对话的尾部**（最后 3-5 轮）
- Agent 主动生成一条"欢迎回来"消息：

> "上次我们聊到了 [摘要]。目前面板上还有 [N] 个假设待验证，[M] 层的证据还不够。你是有新信息要补充，还是想继续上次的方向？"

### 8.3 追加素材

讲师带着新信息回来时（访谈记录、问卷结果、观察笔记），可以：

1. 直接在对话中粘贴 / 上传
2. Agent 读取后，**主动对比面板现有状态**，告知哪些假设被强化/削弱
3. 面板相应更新，变化字段高亮

这个"带新证据回来 → 面板自动重评估"的循环是核心体验之一。

## 九、Agent 行为规格

Agent 的完整 System Prompt 见 `prompts/learning-diagnosis-agent.md`。此处仅列出与产品交互相关的关键规格。

### 9.1 对话人格

**主姿态：共同假设的建构者（80%）**
> "好，如果'执行力差'是当前的工作假设，那我们来想想，哪些情况下这个假设会成立，哪些情况下它可能是错的。"

**辅姿态：温和的盲区指出者（20%）**
> "目前我们讨论了很多'学员不会做'的可能性，但还没有聊到'学员知道怎么做，但选择不做'的可能性。你觉得这个方向值得花两分钟看一下吗？"

### 9.2 单轮回复约束

- 自然语言部分不超过 300 字（除综合建议外）
- 每轮最多提 2 个问题
- 每轮回复必须附带 `<diagnosis_update>` JSON

### 9.3 状态感知行为

Agent 的行为根据诊断状态动态调整（详见 System Prompt 的 DIAGNOSTIC STATES 章节）：

| 状态 | Agent 核心行为 |
|------|---------------|
| COLD_START | 做问题转译，生成竞争假设 |
| HYPOTHESIS_SPARSE | 扩展假设空间，尤其是非培训因素 |
| EVIDENCE_GAP | 提出针对性问题，建议取证行动 |
| EVIDENCE_CONFLICT | 明确冲突，建议判别方法 |
| DECISION_READY | 生成干预建议 |
| USER_EXPLORING | 安静归位信息，不打断 |

### 9.4 取证工具包

当 Agent 建议讲师去收集证据时，不能只说"建议你去了解一下"，而必须提供可操作的工具：

- **迷你访谈提纲**（3-5 个问题，讲师可直接拿去用）
- **脉冲问卷草案**（2-3 分钟完成，可发给学员或经理）
- **观察清单**（讲师如果有机会现场观察时使用）
- **已有材料信号提取**（从讲师之前粘贴的素材中找线索）

## 十一、MVP 范围与分期

### P0：最小可用产品

目标：跑通核心循环——**对话 → 面板生长 → 讲师编辑 → 输出判决书**。

包含：
- [x] 左侧对话区（基本文本输入 + 粘贴长文本）
- [x] 右侧面板：问题转译卡 + 竞争假设区 + 六层诊断板
- [x] 面板渐进生长（阶段一 → 阶段二）
- [x] Agent 每轮更新面板（通过 JSON）
- [x] 面板基础编辑（修改文本、调整假设置信度和状态）
- [x] 面板编辑的系统消息通知（策略 B：轻声确认）
- [x] 冷启动开场白
- [x] 全局证据充分度指示器（红/黄/绿）
- [x] 基础导出（干预判决书 Markdown/PDF）

不包含（推迟到后续版本）：
- [ ] 面板"追问"操作菜单（P1）
- [ ] 方向性编辑的 Agent 主动回应（P1）
- [ ] 快照时间线 / 撤销（P1）
- [ ] 跨会话持久化 / 诊断项目概念（P1）
- [ ] 关键时刻区块（P1）
- [ ] 评估锚点区块（P1）

- [ ] 利益相关方沟通话术生成（P2）

### P1：核心体验完整
