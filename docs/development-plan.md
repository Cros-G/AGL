## 一、开发目标

用 **4 周**（单人全职）完成 P0，交付一个可实际使用的需求诊断工具。

P0 的定义（来自 PRD 第十一章）：**跑通"对话 → 面板生长 → 讲师编辑 → 输出判决书"核心循环。**

P0 完成后讲师可以：
1. 粘贴一段培训需求，看到 Agent 自动做问题转译和假设生成
2. 在对话中持续深入，看到右侧面板逐步长出诊断结构
3. 随时编辑面板中的任何字段，编辑立即生效
4. 看到全局证据充分度指示器（红/黄/绿）
5. 导出一份结构化的干预判决书

## 二、前置决策（开工前必须确认）

PRD 第十二章列出了 8 个开放问题。以下 4 个必须在开工前做出决策，其余可以边做边定。

| # | 问题 | 推荐决策 | 理由 | 何时需要定 |
|---|------|---------|------|-----------|
| 1 | 持久化方案 | **Supabase (托管 PostgreSQL)** | P0 零运维；P1 跨会话持久化直接可用；JSONB 存诊断状态 | 开工前 |
| 2 | Agent API 选择 | **Claude API Messages 直调** | 不需要 Agent SDK 内置工具；需要精细控制流式输出和 JSON 解析 | 开工前 |
| 3 | 面板渲染框架 | **Next.js + Tailwind + shadcn/ui** | 技术方案已选定；shadcn/ui 的 Accordion、Popover、Dialog 可直接用于面板交互 | 开工前 |
| 4 | Context window 管理 | **滑动窗口 + Haiku 压缩** | 技术方案 4.5 节已设计；P0 先实现最近 N 条，摘要压缩在对话超 30 轮时触发 | 第 2 周中期 |

可推迟的决策：
- **#5 多人协作**：P2+ 才考虑，不影响 P0
- **#6 冲突仲裁规则**：P0 已在技术方案中定义了"脏字段追踪 + 冲突弹窗"机制，够用
- **#7 判决书格式标准化**：P0 先出 Markdown，后续课程设计模块启动时再对齐
- **#8 移动端适配**：P0 只做桌面端（≥1024px）

## 四、P0 开发：4 个里程碑

### 里程碑 1：骨架跑通（第 1 周）

**目标**：双面板布局 + 能发消息 + Agent 返回流式文本并显示。

```
用户体验：
  讲师打开页面 → 看到左右双面板 → 在左侧输入文字 →
  Agent 流式回复逐字出现 → 右面板暂时还是空的
```

| # | 任务 | 依赖 | 预估 | 验收标准 |
|---|------|------|------|---------|
| 1.1 | **双面板布局** | 环境搭建 | 3h | 左右面板 45/55 分割显示；分隔线可拖拽；右面板可折叠 |
| 1.2 | **顶栏** | 1.1 | 1.5h | 项目名称（可编辑）+ 证据充分度指示器占位（灰色，暂不活动） |
| 1.3 | **消息输入区** | 1.1 | 2h | 底部输入框，支持多行、粘贴长文本、Enter 发送、Shift+Enter 换行；发送按钮有/无内容两种状态 |
| 1.4 | **消息列表（静态渲染）** | 1.1 | 2h | 用户消息右对齐蓝底、Agent 消息左对齐白底描边、系统消息居中灰色，全部遵循 design-system 规范 |
| 1.5 | **后端：消息 API + Agent Orchestrator 骨架** | 环境搭建 | 4h | `POST /api/projects/{id}/messages` 接收消息，调用 Claude API，返回 SSE 流 |
| 1.6 | **Agent 上下文组装（基础版）** | 1.5 | 3h | System Prompt 加载 + 完整对话历史注入（暂不做压缩）+ 新消息 |
| 1.7 | **流式显示** | 1.4, 1.5 | 3h | 前端消费 SSE 流，Agent 回复逐字显示，末尾闪烁光标；完成后光标消失 |

**M1 验收**：打开页面，粘贴一段培训需求，Agent 流式回复，刷新后对话不丢失。

| 2.4 | **问题转译卡组件** | 2.3 | 3h | 四个字段 + 叙事偏差警告条；遵循 design-system 6.5 节规范；字段更新时有高亮动画 |

### 里程碑 3：编辑闭环（第 3 周）

**目标**：面板上所有字段可编辑；编辑通过系统消息同步给 Agent；冲突可检测。

用户体验：
  讲师点击面板中的假设内容 → 原地编辑 → 保存后对话区出现灰色系统消息 →

## 八、风险管理

### 8.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 | 触发时机 |
|------|------|------|---------|---------|
| **Agent 输出的 JSON 格式不稳定** | 高 | 面板更新失败 | Zod partial 验证 + 降级（只应用合法字段）+ System Prompt 中强调格式要求 | M2 阶段（2.1 响应解析） |
| **长对话超出 context window** | 中 | Agent 丧失上下文 | Context Window Manager + Haiku 摘要压缩 | M4 阶段（4.5） |
| **SSE 流中断（网络波动）** | 中 | 响应不完整 | 前端检测断连 → 自动重试；消息 ID 去重防止重复 | M4 阶段（4.6） |
| **面板编辑与 Agent 更新竞态** | 中 | 数据不一致 | 脏字段追踪 + 冲突弹窗 | M3 阶段（3.9-3.10） |
| **Claude API 延迟波动** | 低 | 用户等待时间长 | 流式输出让用户不必等完整响应；loading 状态给予反馈 | 全程 |

### 8.2 产品风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| **讲师不理解"竞争假设"概念** | 中 | 核心功能被忽略 | 开场白中自然引导；假设卡片设计足够直觉；考虑加入 onboarding tooltip |
| **Agent 生成的假设质量不够** | 中 | 信任崩塌 | 持续迭代 System Prompt；在场景测试中收集 bad cases 逐条修正 |
| **面板信息密度过高** | 低 | 讲师认知过载 | 渐进生长机制 + 折叠/展开 + 只在相关维度出现时才显示 |
| **"不做培训"建议在组织中不被接受** | 高 | 讲师不敢用这个输出 | P2 加入利益相关方沟通话术；判决书中把"不做培训"包装为专业判断 |

### 8.3 风险应对的检查点

| 时间 | 检查项 | 如果不通过 |
|------|--------|-----------|
| M2 完成时 | Agent 的 JSON 输出在 10 次测试中至少 8 次通过 Zod 验证 | 停下来优化 System Prompt 的格式指令，增加 few-shot examples |
| M3 完成时 | 3 个真实培训场景中，面板至少能正确显示 2 个场景的假设和诊断层 | 审查 Agent 的状态判断和假设生成逻辑 |
| M4 完成时 | 非开发者（讲师或产品人员）能在 5 分钟内完成一次冷启动到首次面板更新 | 回退到更简单的冷启动流程 |

## 九、测试策略

### 9.1 P0 测试范围

P0 不做自动化测试（ROI 不足），以手动场景测试为主。

| 层级 | 覆盖内容 | 方式 |
|------|---------|------|
| **Agent 输出质量** | System Prompt 在不同输入下的表现 | 准备 5 个典型培训需求作为测试用例，逐个验证 Agent 的转译/假设/状态判断质量 |
| **JSON 可靠性** | 解析成功率、降级行为 | 在每个测试场景中记录 JSON 通过率，低于 80% 则优化 prompt |
| **面板渲染** | 所有区块在各状态下正确显示 | 遍历面板的每个组件，覆盖正常态/空态/满载态/已排除态 |
| **编辑闭环** | 编辑→系统消息→Agent 感知 | 在对话中途编辑假设，验证 Agent 下轮回复是否引用了修改后的内容 |
| **冲突处理** | 编辑+Agent 回复竞态 | 手动模拟：编辑一个字段后立即发消息，验证冲突弹窗正确弹出 |
| **导出** | 判决书内容完整性 | 导出后检查每个章节是否有内容，null 字段是否标注"待确认" |

### 9.2 测试用例（5 个场景）

| # | 场景 | 测试重点 |
|---|------|---------|
| 1 | "给店长做执行力培训" | 冷启动→转译→假设生成→面板种子阶段完整流程 |
| 2 | "销售不会讲方案价值，需要表达培训" | 多轮对话→面板从种子到分枝→六层诊断板逐层出现 |
| 3 | "新员工入职培训效果不好" | Agent 能识别出非培训因素（环境/管理）→"不建议培训"路径 |
| 4 | 讲师粘贴一长段业务方微信消息 | 长文本输入→Agent 正确解析→转译质量 |
| 5 | 对话 20 轮以上的深度诊断 | Context window 管理→Agent 不丧失上下文→面板状态一致性 |

当前所有文档都在 `prompts/` 下，但它们不全是 prompt。建议调整为：

```
AGL/
├── docs/
│   ├── PRD-learning-diagnosis.md          ← 产品需求文档
│   ├── tech-design-learning-diagnosis.md  ← 技术方案

│
├── prompts/
│   └── learning-diagnosis-agent.md        ← Agent System Prompt（唯一真正的 prompt）
│
├── materials/
│   ├── learning_diagnosis.md              ← 领域洞察原始材料
│   └── GUI_AGL.md                         ← GUI vs Agent 分析原始材料
│
└── src/                                   ← 开发启动后的代码目录
    └── ...
```

只有 `learning-diagnosis-agent.md` 是真正的 prompt 文件，其余都是产品/技术/设计文档，应归入 `docs/`。
